# Contributing

## 1. Welcome and Scope

VibeGear2 is an original open-source arcade road racer for the web. Start
with [`README.md`](../README.md), [`GDD.md`](../GDD.md), and the Markdown GDD
tree under [`docs/gdd/`](gdd/). The project is built to accept community code,
data, art, audio, and modding work, provided each contribution follows the
workflow and originality rules below. For mod-specific rules, read
[`MODDING.md`](MODDING.md) before changing any mod data or loader behavior.

## 2. Code of Conduct

The full code of conduct is still TBD. Until that lands, treat other
contributors professionally, assume good faith, and keep critique focused on
the work. Escalate moderation issues to maintainers in GitHub Issues.

## 3. Branch and PR Workflow

Follow [`WORKING_AGREEMENT.md`](WORKING_AGREEMENT.md) sections 2, 3, and 4.
Create short-lived branches from `main` named `feat/<slice>`,
`fix/<slice>`, `chore/<slice>`, or `docs/<slice>`. Keep one slice per PR.
Never push directly to `main`. Each PR should link the relevant GDD section,
the matching progress-log entry, the test plan, and any followups or open
questions created by the slice.

## 4. Commit Message Style

Use this format:

```text
<type>(<area>): <imperative summary>

<why this change exists, not what it does>
```

Allowed types are `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, and
`perf`. Keep the subject imperative and concrete.

## 5. Verification Before Opening a PR

Run the baseline local check before opening a PR:

```sh
npm run verify
```

`npm run verify` covers lint, typecheck, unit tests, and content lint.
PRs that touch gameplay, UI, persistence, routing, rendering, or deployment
should also run Playwright:

```sh
npm run test:e2e
```

CI reruns the matching checks on every push.

## 6. PR Confirmation Checklist

Copy this checklist into PRs that add code, content, data, art, audio, or
docs:

- [ ] Original work, or work with a compatible licence noted in the asset
  manifest.
- [ ] No Top Gear 2 assets, names, soundalikes, layouts, rips, or
  trademark-risk content. See [`LEGAL_SAFETY.md`](LEGAL_SAFETY.md).
- [ ] Any third-party source documented in the asset manifest.
- [ ] All new content and data passes `npm run verify`.
- [ ] No em-dashes or en-dashes in any touched text.
- [ ] `PROGRESS_LOG.md` entry added if the slice is structural.

## 7. Originality Requirement

Original work means one of:

- Authored from scratch by the contributor.
- Based on CC0, CC BY 4.0, or CC BY-SA 4.0 material with provenance and
  attribution documented.
- Procedurally generated by a contributor-owned process, with the generation
  source or notes retained when practical.

Do not copy or trace Top Gear 2 assets, other commercial game assets, real car
badges, real track branding, copyrighted music, ROM data, or unclear sample
packs. [`LEGAL_SAFETY.md`](LEGAL_SAFETY.md) is the canonical safe and unsafe
pattern catalogue.

## 8. Licensing Model

Source code is MIT licensed. See [`LICENSE`](../LICENSE). Original art,
sprites, sound effects, music, and asset manifests default to CC BY 4.0. See
[`ASSETS-LICENSE`](../ASSETS-LICENSE). Track, championship, balancing, and
community mod data default to CC BY-SA 4.0. See
[`DATA-LICENSE`](../DATA-LICENSE).

By opening a PR, contributors agree that their contribution is licensed under
the matching project licence. The project uses inbound equals outbound:
no CLA and no DCO sign-off are required. Relicensing requires unanimous
contributor consent.

## 9. Asset Manifest Rules

Every art contribution must update [`public/art.manifest.json`](../public/art.manifest.json)
with the required manifest fields: `id`, `path`, `kind`, `license`,
`source`, `originality`, and `date`. Follow
[`LEGAL_SAFETY.md`](LEGAL_SAFETY.md) section 6 for the canonical field
requirements. Audio, track, championship, balancing, and mod-data
contributions must include the corresponding required manifest metadata in
the matching manifest once that content loader exists. Until then, include
the metadata beside the data change in the PR description and ask for
`legal-review` if the source is not fully original. A PR that ships binary
assets or third-party data without manifest metadata will not merge.

## 10. Schema and Lint Expectations

Data files must validate against the relevant Zod schema in
[`src/data/schemas.ts`](../src/data/schemas.ts). Track, car, upgrade, AI,
championship, save, leaderboard, and future mod-manifest data should be
schema-first. Lint warnings block merge. Content-lint violations block merge.
Do not bypass checks with `--no-verify`.

For setup, use the reproducible install flow in [`LOCAL_DEV.md`](LOCAL_DEV.md)
with `npm ci`. Use `npm run verify` to run the full validation suite for code,
docs, art, and audio checks. For a complete command reference, read
[`SCRIPTS.md`](SCRIPTS.md).

## 11. Issue Triage and Labels

Use these labels when filing issues or triaging PRs:

- `physics`
- `renderer`
- `ai`
- `ui-ux`
- `audio`
- `modding`
- `content`
- `legal-review`
- `good-first-issue`
- `help-wanted`
- `performance`
- `bug`
- `design`

Use `legal-review` for originality, licence, trademark, or source-provenance
questions. Maintainers may block merge until the legal-review question is
resolved.

The canonical label definitions live in [`.github/labels.yml`](../.github/labels.yml)
and are synced on pushes to `main` by [`.github/workflows/labels.yml`](../.github/workflows/labels.yml).
Use the issue templates under [`.github/ISSUE_TEMPLATE/`](../.github/ISSUE_TEMPLATE/)
for bug reports, GDD-grounded design discussions, and contributor-friendly
starter tasks.

## 12. First-Time Contributor Walkthrough

1. Clone the repo and install dependencies:

   ```sh
   npm install
   npx playwright install chromium
   npm run dev
   ```

2. Read [`AGENTS.md`](../AGENTS.md), [`WORKING_AGREEMENT.md`](WORKING_AGREEMENT.md),
   [`GDD.md`](../GDD.md), and [`LEGAL_SAFETY.md`](LEGAL_SAFETY.md).
3. Pick a `good-first-issue`, create a branch from `main`, make the smallest
   useful slice, run verification, and open a draft PR.

## 13. Maintainer Expectations

Maintainers review for correctness, scope control, originality, licence
metadata, tests, and consistency with the GDD. Review timing is TBD, but the
project expects maintainers to respond with concrete requested changes rather
than vague rejection. Maintainers squash-merge by default once CI is green and
review is satisfied.

## 14. Where to Ask Questions

Use GitHub Issues with `help-wanted` until Discussions is enabled. Do not
paste copyrighted assets, ROM data, ripped audio, or reference screenshots
from other games into issues, PRs, docs, or the wiki. Use a text description
or an original diagram instead.

## Renderer Performance Notes

Run the render bench locally before any PR that touches the renderer
(parallax, dust, vfx, sprite atlas, the strip drawer, or the segment
projector):

```sh
npm run bench:render
```

The bench is informational. It is not a CI gate because frame-time numbers are
machine-dependent. Paste the output in renderer PRs so reviewers can compare
relative changes on the same machine.

## House Style

Do not use em-dashes or en-dashes anywhere. Before committing docs or
user-facing text, run:

```sh
perl -ne 'print "$ARGV:$.:$_" if /[\x{2013}\x{2014}]/' docs/*.md docs/gdd/*.md README.md
```

The command must return no matches.
